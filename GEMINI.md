# GEMINI.md - Adlar Castra Heatpump Project Context

This document provides foundational mandates and project-specific instructions for the Adlar Castra Heatpump Homey app. These instructions take absolute precedence over general defaults.

## 🚀 Project Overview
- **Project Name:** `org.hhi.adlar-heatpump` (Adlar Castra Heatpump)
- **Goal:** Local integration of Adlar Castra Aurora series heat pumps with Homey Pro via Tuya Local API.
- **Technologies:** TypeScript (ES2020), Node.js, Homey SDK v3, TuyAPI (customized).
- **Architecture:** **Service-Oriented Architecture (SOA)**. A central `ServiceCoordinator` orchestrates 9 specialized services (TuyaConnection, AdaptiveControl, COPCalculator, FlowCardManager, etc.).

## 🚨 Core Mandates

### 1. Configuration Management (CRITICAL)
- **NEVER manually edit `app.json`.** This file is auto-generated by Homey Compose.
- **ALWAYS edit `.homeycompose/app.json`** or the modular files in `.homeycompose/capabilities/`, `.homeycompose/flow/`, and `.homeycompose/drivers/`.
- Manual changes to `app.json` will be overwritten and lost during the build process.

### 2. SVG Icon Guidelines (iOS/Safari Compatibility)
- WebKit (iOS) fails to inherit styling from the root `<svg>` element.
- **FORBIDDEN:** Attributes (`fill`, `stroke`, `stroke-width`, etc.) on the root `<svg>` element.
- **REQUIRED:** Every shape element (`<path>`, `<rect>`, `<circle>`, etc.) MUST have explicit `fill="currentColor"` and `stroke="currentColor"` (or `fill="none"`).
- Root `<svg>` should ONLY contain `viewBox="0 0 100 100"` and `xmlns`.

### 3. Agent & Git Authorization Policy
- **ANALYZE FIRST:** Document findings and proposed solutions. Wait for explicit user approval before modifying code.
- **GIT RESTRICTIONS:** Only `github-release-manager` is authorized for `git commit`, `git tag`, and `git push`.
- **MAIN ASSISTANT:** ONLY commit when explicitly requested ("commit this"). ALWAYS show what will be committed first.

## 🏗 Engineering Standards

### Timers & Cleanup
- **ALWAYS use `this.device.homey.setTimeout()` / `setInterval()`** (or `this.homey.setTimeout()`).
- **NEVER use global `setTimeout()` / `setInterval()`**; Homey's timers ensure automatic cleanup and prevent memory leaks.
- **Cleanup:** Clear all timers in `onDeleted()` or `destroy()` paths.

### Logging (v2.1.0+ Structured Logger)
- **NEVER use `console.log()`.**
- **ALWAYS use the structured `Logger`** instance with levels: `ERROR`, `WARN`, `INFO`, `DEBUG`.
- In `device.ts`, the logger is initialized via `onInit()`.

### Type Safety
- **NEVER use `as any`.**
- Prefer `@ts-expect-error` with an explanatory comment when accessing library internals.

### Event Handlers (TuyAPI)
- Keep event handlers **synchronous**.
- If async work is required, use a "fire-and-forget" pattern with `.catch()` to prevent unhandled rejections.

## 🗺 Project Architecture (SOA)

The app is divided into specialized services managed by `lib/services/service-coordinator.ts`:

1.  **TuyaConnectionService:** Low-level TuyAPI communication, heartbeat, and reconnection logic.
2.  **AdaptiveControlService:** PI-based temperature regulation and building model learning.
3.  **COPCalculator / RollingCOPCalculator / SCOPCalculator:** Efficiency analysis (real-time, time-series, and seasonal).
4.  **FlowCardManagerService:** Dynamic registration of 90+ flow cards based on device health.
5.  **SettingsManagerService:** Prevents race conditions and handles complex device settings.
6.  **BuildingInsightsService:** Generates energy-saving recommendations based on learned data.
7.  **EnergyTrackingService:** Integrates external power/energy data.

### Key Files
- `drivers/intelligent-heat-pump/device.ts`: The main device class (delegates to services).
- `lib/definitions/adlar-mapping.ts`: Maps Tuya DPS IDs to Homey capabilities.
- `lib/constants.ts`: Global thresholds and configuration.
- `lib/error-types.ts`: Centralized error categorization and recovery.
